Vaelix OS

My own Linux distro. I got tired of undoing Ubuntu's defaults every fresh install, so I built one that starts where I actually work. 13 rebuilds before it booted clean.
→

Why this exists

Every fresh Ubuntu install means the same thing — undo the defaults I hate, fix the package conflicts, rebuild my environment from zero. I got tired of that cycle, so I built my own starting point.

How I built it

Ubuntu 24.04 LTS base. KDE Plasma 6 — the only DE that's actually scriptable without fighting it. XanMod kernel with BORE scheduler, BBR2 TCP. The 13 rebuilds weren't all dramatic failures. Most were boring config drift between kernel and DE that only showed after first login. The real lesson was simpler: lock your kernel minor version before you spend a week debugging display issues.

Where it is now

v1.1 "Titanium Velocity" is stable — it's my daily driver on the LOQ. v2.0 "Renaissance" is in progress: full Wayland, Rust for core tooling, GrapheneOS-style hardening, Kali sandbox in Podman.

What I'd do differently

Write down every failure the moment it happens. I know the 13-rebuilds story by heart but I don't have the specific reasons documented. That's the part that actually matters and I lost it.

LOQ Control Center

I switched my laptop to Linux and realized there was no hardware control panel. So I wrote one. What started as a midnight GPU-switching script became a 6-release, MIT-licensed tool running on production machines.
→

Why this exists

I switched to Linux with no dual boot and no safety net. Everything was great until I realized I had zero control over fan profiles, power modes, or GPU switching — things Lenovo Vantage handled for free on Windows. I spent a week reading sysfs docs and then just started writing it.

The v1 → v2 story

v1 wrote directly to ACPI sysfs through pkexec. That meant every profile switch launched an auth dialog that opened behind the main window — invisible to the user. Hardware writes would fail silently, UI would freeze, and nothing told you why. I redesigned it from scratch: v2 talks to power-profiles-daemon over D-Bus as a normal user. No root. No auth dialogs. Profile switching is instant.

What v2 actually does

Four power profiles matching the LOQ hardware buttons — Quiet, Balanced, Performance, Custom. CPU PL1/PL2 via RAPL, GPU TGP via nvidia-smi, fan curve editor rendered with Cairo. Fn+Q keyboard sync via a 1-second sysfs poll so the hardware button and the UI stay in sync. Rewrote the sensor stack to read directly from hwmon — this alone killed ~300 noisy stderr lines per minute.

What I learned

The difference between code that works on your machine and software someone else can rely on. Also that privilege escalation on Linux is more nuanced than just sudo — D-Bus and Polkit exist for a reason.

thermal.pyPython
def apply_thermal_policy(self, profile: ThermalProfile) -> None:
    """Apply thermal policy via Polkit — no root needed."""
    try:
        proxy = self.bus.get_object(
            'com.vaibhav.loqcontrol',
            '/com/vaibhav/loqcontrol/Thermal'
        )
        iface = dbus.Interface(proxy, 'com.vaibhav.loqcontrol.Thermal')
        iface.SetProfile(profile.name, profile.pl1, profile.pl2)
    except dbus.DBusException as e:
        if 'NotAuthorized' in str(e):
            raise PermissionError("pkexec auth required")
        raise

Android 13 Bring-Up — Moto G5 Plus

Bringing up LineageOS 20 on hardware eight years behind mainline. Blocked on bootloader trust chain. Three months in — still going. That's the point.
→

Why this exists

I wanted to stop assuming and start knowing how the Android kernel actually talks to hardware. The Moto G5 Plus (potter) runs Qualcomm MSM8953 on Linux 3.18 — 8 years behind mainline. No maintained LineageOS support. No public docs. Perfect.

How I'm doing it

Methodical layers: vendor blobs → kernel patches → HAL interfaces. Building device tree, vendor tree, kernel patches, and OrangeFox recovery tree in parallel. Completed two kernel builds (~15.29 MiB each), resolved rootfs flashing, SELinux, and display driver config. Current state: boots to fastboot, doesn't pass verified boot.

What I'm learning

Android build system at every layer — Soong, Make, Blueprint. How Treble-era abstractions fail on pre-Treble hardware. What "device can't be trusted" means at bootloader level, and the four different ways to address it.

Currently blocked on

Bootloader trust chain. Spent two weeks thinking this was a key signing problem. It isn't — it's a verified boot chain issue. The bootloader checks signatures on the boot image, but the OEM hash of the device tree doesn't match what LineageOS expects. Trying test key signing via avbtool first. If that fails, custom bootloader flash — risky since Motorola's tool doesn't expose enough control.

ScholarSync

24-hour hackathon build. AI study tool that answers across your whole library, not just the file you have open. Won Google GDSC TechSprint.
→

Why this exists

The way students study is genuinely broken. You have 10 PDFs open, you're switching between them, and the mental overhead of connecting it all is the thing that blocks understanding — not the content itself. Most AI tools answer one document at a time. That's not how studying actually works.

How I built it

24-hour Google GDSC TechSprint. Decoupled from the start — Next.js 14 App Router, FastAPI, ChromaDB, Gemini 2.0 Flash. Multi-doc RAG with page-level citations, interactive knowledge graph, 3D CSS flashcards. Went with batch responses over streaming because getting SSE right in 24 hours while also building the graph and flashcard system wasn't realistic.

What I learned

How RAG actually works end to end — chunking, embeddings, retrieval, structured output. Also that citations matter more than confident-sounding answers when you're building a study tool. An AI that admits what page it got something from is more trustworthy than one that sounds sure of itself.

What I'd change

Streaming from day one — the 4-8 second wait on large queries is noticeable. And the knowledge graph should let you explore nodes interactively instead of firing another LLM call. Both are on the list.

2026-06-28
LOQ v2: separating the UI from hardware calls was the fix
v1 would write directly to sysfs through pkexec every time you switched a profile. That spawned an auth dialog that opened behind the main window. Users couldn't see it, hardware writes failed silently, and the UI froze. I sat with that for a while before I saw the fix: push hardware control into a systemd daemon and talk to it over D-Bus as a normal user. No pkexec. No dialog. Profiles switch immediately, and the UI never blocks.

Vaelix v1.1: what actually caused the 13 rebuilds
Rebuilds 1–3: kernel version incompatibility with SDDM's DRM interface. Rebuilds 4–6: XanMod flags that silently break hibernation on Intel + NVMe power states. Rebuilds 7–10: Cairo/Pango font rendering mismatch that only showed after first login. Rebuilds 11–13: sysfs power management differences between kernel minor versions. Lock your kernel minor version before you touch anything else. That's the lesson. I learned it the hard way across 13 builds.

ScholarSync: batch was the right call for 24 hours
Streaming Gemini through FastAPI means SSE or WebSocket on both ends. Getting that working properly while also building the knowledge graph, the flashcard system, and the RAG pipeline in one night wasn't going to happen. Batch responses meant 4-8 second waits on heavy queries, but they were stable, debuggable, and we shipped. We won. Streaming is the obvious next step and it's coming — but the call to go batch was right for that window.
