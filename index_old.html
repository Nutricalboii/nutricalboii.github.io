LOQ Control Center

The Ultimate Hardware Control Suite for Lenovo LOQ Laptops on Linux.

The problem

Linux users with Lenovo LOQ laptops had no unified way to manage thermal profiles, GPU switching, or power limits. The options were fragmented scripts, high-risk sudo commands, or directly manipulating the Embedded Controller. Every mistake carried hardware-damage risk. The experience was unacceptable.

The approach

Started as a GPU switching script written at midnight because Linux took something that should have worked and didn't. It evolved over 6 versioned releases into a production-grade orchestration platform: a systemd daemon, an adaptive thermal engine, a Polkit-gated privilege architecture that never needs full root, and a full GTK4 GUI with interactive 7-point fan curves rendered with Cairo.

What I learned

Hardware management at the Linux kernel level — ACPI interfaces, Embedded Controller communication, sysfs power nodes. Security-first privilege escalation patterns with pkexec instead of raw sudo. GTK4 application design. The difference between a script that works once and software that other people can depend on.

What I'd change

The initial architecture mixed UI logic with hardware interfaces. V2 separated them cleanly, but I'd start that way from day one. Also, user-facing error messages could be significantly better — most still surface raw hardware codes.

ScholarSync

AI-powered "Second Brain" for students — breaking information silos through multimodal RAG.

The problem

Students drown in disconnected PDFs. Each document is an island. Drawing connections between chapters, textbooks, or subjects requires the kind of mental overhead that blocks actual understanding. Study tools existed, but none synthesized across an entire library simultaneously.

The approach

Built for TechSprint Hackathon. Chose a fully decoupled architecture: Next.js 14 frontend, FastAPI backend, ChromaDB vector store. Gemini 2.0 Flash handles multi-document synthesis through RAG — taking a user's entire library and answering questions that cite specific files and page numbers. Interactive knowledge graphs visualize concept relationships. 3D CSS flashcard animations make active recall feel tactile rather than clinical.

What I learned

Multimodal RAG pipeline design — chunking, embedding, semantic search, and structured JSON output from LLMs. How to make AI output feel trustworthy through citation rather than confident assertion. The performance tradeoffs between vector search accuracy and query latency.

What I'd change

The graph rendering is good but could be more interactive — clicking a node should open an exploration panel, not just an AI dive. Also would add streaming responses from the start; the current batch approach feels slow on large queries.

Android 13 Bring-Up — Moto G5 Plus

LineageOS 20.0 bring-up on MSM8953 / Linux 3.18 kernel. Three months in. Still going. That's the point.

The problem

The Moto G5 Plus (codename: potter) runs a Qualcomm MSM8953 SoC on a Linux 3.18 kernel — over 8 years behind mainline. Getting Android 13 to boot on hardware that old means negotiating between an ancient vendor tree, a modern Android build system, and driver interfaces that simply didn't exist in 2017.

The approach

Methodical layers: vendor blobs first, then kernel bring-up patches, then HAL-level interfaces. I've been building the device tree, vendor tree, kernel patches, and OrangeFox recovery tree in parallel. Current state: device boots to fastboot, system doesn't pass verified boot. The Halium overlay isn't yet trusted by the bootloader.

What I'm learning

How the Android build system works at every layer — Soong, Make, Blueprint. How the kernel talks to HALs. How Treble-era abstractions fail on pre-Treble hardware. What "device can't be trusted" actually means at the bootloader level, and the four different ways to address it.

Currently blocked on

Bootloader trust chain. Exploring signature bypass via test keys, dm-verity disabling, and custom bootloader flash. This is the hard part — which is why I'm here.

Repos in this project

  • android_device_motorola_potter (C++)
  • android_vendor_motorola_potter (Makefile)
  • android_kernel_motorola_msm8953 (C)
  • android_build_patches_potter (Shell)
  • orangefox_potter_tree (Makefile)
  • android13 (C++ — base porting work)

This project exists because I wanted to stop assuming and start knowing how the kernel actually talks to hardware. Three months in, I understand things about Android internals that no tutorial would have shown me.

Avora

Minimalist enterprise SaaS landing with a fully architected dashboard and AI microservice backend.

The approach

Decoupled architecture from the start: public-facing Next.js landing page pushing leads directly to Google Sheets via Apps Script (zero-config for launch), with a full enterprise dashboard on protected routes backed by MongoDB + Prisma. A separate Python FastAPI microservice scaffolded for future AI workloads. 59 commits. Deployed on Vercel.

What I learned

How to architect a product that can launch small and scale without rewriting. Next.js App Router Server Actions. The tradeoff between launch velocity and infrastructure debt. When Google Sheets is genuinely the right database for early-stage lead capture.
